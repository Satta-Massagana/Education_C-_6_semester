# MPI-подобный обмен сообщениями для локального кластера

Проект реализует механизм обмена сообщениями между узлами кластера без Microsoft MPI, OpenMPI, Akka.NET, Orleans и других готовых фреймворков распределенных вычислений.

## Состав файлов

- `Program.cs` — точка входа, создание локального кластера, запуск тестов и вывод результатов.
- `MpiCommunicator.cs` — MPI-подобный интерфейс: `Send`, `Receive`, `SendReceive`, `Broadcast`, `Gather`, `Scatter`, `AllGather`, `Reduce`, `AllReduce`.
- `MpiNode.cs` — TCP-узел с `TcpListener`, подключениями к другим узлам и маршрутизацией сообщений.
- `MessageTypes.cs` — типы сообщений, теги, структура `MpiMessage`, сообщения точка-точка и коллективные сообщения.
- `CollectiveOperations.cs` — реализации Broadcast, Gather, Scatter, Reduce и Barrier.
- `MpiBenchmark.cs` — измерение задержки, времени выполнения и пропускной способности операций.

## Как работает реализация

Программа создает несколько `MpiNode` на локальной машине. Каждый узел слушает свой порт: `9000`, `9001`, `9002`, `9003` для основного теста из 4 узлов. Затем узлы соединяются в полную mesh-топологию через `TcpClient`.

Каждое сообщение передается по TCP в формате:

1. 4 байта длины сообщения.
2. JSON-представление `MpiMessage`.
3. Полезная нагрузка внутри `Payload` как `byte[]`.

Для фильтрации используются `SourceRank`, `MessageType` и `Tag`. Все операции ожидания имеют таймаут, чтобы программа не зависала бесконечно при ошибке соединения или нарушении порядка вызовов.

## Коллективные операции

- `Broadcast` использует бинарное дерево: root отправляет сообщение максимум двум узлам, а промежуточные узлы пересылают его дальше.
- `Gather` использует дерево сбора к root: промежуточные узлы агрегируют данные своего поддерева и отправляют один пакет родителю.
- `AllGather` использует кольцевую топологию: каждый узел за шаг передает один фрагмент следующему узлу.
- `Scatter` использует дерево: узел передает потомку только ту часть данных, которая нужна этому поддереву.
- `Reduce` использует бинарное дерево снизу вверх: дочерние узлы отправляют агрегированные значения родителю.
- `Barrier` состоит из двух фаз: сбор `READY` на root и рассылка `GO` по дереву.

## Запуск

```
dotnet run
```

Программа выводит:

- задержку point-to-point обмена;
- время и количество сообщений для Broadcast;
- время и объем данных для Gather;
- результаты Reduce для суммы, максимума и минимума;
- время Barrier;
- масштабируемость Broadcast, Gather, Reduce и Barrier для 2, 4, 8 и 16 узлов;
- единый итоговый вывод по операциям: участники обмена, объем данных, число сообщений, среднее время полной операции и пропускную способность.
- раздел `Сравнение`, где Broadcast, Gather и Reduce дополнительно нормализуются как время полной операции, деленное на число узлов-получателей или узлов-источников.

Перед измерениями выполняется warm-up: несколько служебных операций без учета времени, чтобы прогреть JIT, ThreadPool, TCP-циклы чтения и сериализацию.

В блоке масштабируемости миллисекунды считаются на отдельном локальном кластере заданного размера. Для каждого размера кластера программа создает новые узлы, выполняет warm-up, запускает операцию 20 раз и выводит `average`, `min`, `max` и `median`.

Количество измеряемых запусков задается одной константой в `Program.cs`: сейчас это 20 повторений для основных тестов и для блока масштабируемости. Для `Reduce` один повтор включает три редукции: сумма, максимум и минимум. Warm-up выполняется отдельно и в эти числа не входит.

Фактические числа зависят от нагрузки машины, версии .NET и состояния TCP-портов. Для повторного запуска желательно дождаться освобождения портов или изменить базовые порты в `Program.cs`.
