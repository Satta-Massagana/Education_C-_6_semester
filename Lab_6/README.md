# Лабораторная работа: атомарные операции и lock-free структуры

## Состав решения

| Файл | Назначение |
|------|------------|
| `AtomicCounter.cs` | Счётчик на `Interlocked` без блокировок |
| `LockFreeStack.cs` | Стек Трайбера на `Interlocked.CompareExchange` |
| `StatisticsTracker.cs` | Статистика запросов на атомарных приращениях |
| `AtomicBenchmark.cs` | Замеры: Interlocked vs `lock`, нагрузка на стек |
| `Program.cs` | Точка входа, сценарии нагрузки и вывод отчёта |

## Реализация

- **`AtomicCounter`**: поле `long`, все изменения через `Interlocked.Increment`, `Decrement`, `Add`, `Exchange`, `CompareExchange`, `Read`; `GetAndIncrement` / `GetAndDecrement` построены на инкременте/декременте с поправкой на возвращаемое значение API.
- **`LockFreeStack<T>`**: узлы `Node` с полями `Value` и `Next`; голова `_head` обновляется только CAS-циклами; чтение головы выполняется через `CompareExchange(ref _head, null, null)` без отдельной «синхронизации на volatile».
- **`StatisticsTracker`**: четыре счётчика `long`, обновление через `Increment`/`Add`, сброс через `Exchange`.
- **`AtomicBenchmark`**: эталон с `lock` вынесен во вложенный `LockedCounter` только для сравнения; lock-free стек сначала наполняется параллельно, затем параллельно опустошается с повторными попытками `TryPop`, пока стек не станет пустым.

## Ожидаемые результаты нагрузочного сравнения

Конкретные цифры зависят от процессора, числа ядер, профиля питания ОС и версии .NET. Обычно на конкурентных инкрементах/декрементах вариант с **`Interlocked` быстрее**, чем общий `lock` на одном объекте, а коэффициент ускорения растёт с числом потоков из‑за отсутствия сериализации на критической секции.

Показатель **«накладные расходы блокировок»** в программе считается как \((T_{\mathrm{lock}} - T_{\mathrm{interlocked}}) / T_{\mathrm{lock}} \cdot 100\%\) при положительном числителе это доля времени, «съеденная» блокировкой относительно замера с `lock`.

## Запуск

Из каталога проекта:

```
dotnet run
```

Будут выведены блок **«Результаты тестирования…»**, сообщение о целостности синхронизации и блок **«Сравнение атомарных операций и блокировок»** из `AtomicBenchmark.CompareAllApproaches()`.

## Примечание по ABA

Классический стек на CAS уязвим к **ABA-проблеме**, если узлы переиспользуюсь из пула; здесь узлы только создаются при `Push` и не возвращаются в пул, поэтому для учебной модели ABA не моделируется. В промышленном коде для указателей на память часто используют tagged pointers / hazard pointers / epoch reclamation.
