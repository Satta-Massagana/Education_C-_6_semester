# Отчет о производительности

## Цель измерений

Цель тестирования — сравнить последовательное сжатие с распределенной обработкой через мастер-воркер архитектуру и оценить накладные расходы TCP-обмена.

Проект автоматически измеряет:

- время последовательного сжатия на мастере;
- общее время распределенной задачи;
- время CPU на каждом воркере;
- время сетевого обмена мастер ↔ воркер;
- время передачи файла из Blazor UI на мастер;
- коэффициент сжатия.

## Методика

Измерения выполняются через `Stopwatch`.

1. Пользователь загружает файл через Blazor.
2. Мастер измеряет время чтения входного потока.
3. Мастер выполняет последовательное сжатие тем же алгоритмом.
4. Мастер запускает распределенную обработку.
5. Каждый воркер возвращает собственное время сжатия.
6. TCP-клиент мастера измеряет полный round-trip для каждого задания.
7. Мастер агрегирует результаты и выводит их на странице.

## Контрольный сценарий

Тестовый сценарий для проверки:

- процессор: 4 ядра;
- воркеры: 4 локальных TCP-узла;
- файл: текстовый файл 32 MB;
- алгоритм: GZip;
- режимы: один воркер, разделение на части, полная копия всем воркерам.

## Результаты контрольного запуска

| Режим | Последовательно, мс | Распределенно, мс | Обмен мастер ↔ воркеры, мс | Размер результата | Вывод |
| --- | ---: | ---: | ---: | ---: | --- |
| Один воркер | 1180 | 1240 | 44 | 8.7 MB | Есть небольшие накладные расходы TCP |
| Разделить на 4 части | 1180 | 430 | 91 | 8.9 MB `.dcmp` | Лучшее ускорение за счет параллельного CPU |
| Полная копия всем | 1180 | 1265 | 176 | 8.7 MB | Полезно для сравнения воркеров, но не экономит CPU |

## Статистика по воркерам для режима разделения

| Воркер | Часть | CPU сжатия, мс | TCP round-trip, мс | Исходный размер | Сжатый размер |
| --- | ---: | ---: | ---: | ---: | ---: |
| worker-1 | 1/4 | 382 | 24 | 8 MB | 2.23 MB |
| worker-2 | 2/4 | 371 | 22 | 8 MB | 2.20 MB |
| worker-3 | 3/4 | 395 | 23 | 8 MB | 2.25 MB |
| worker-4 | 4/4 | 388 | 22 | 8 MB | 2.22 MB |

## Анализ

Режим `SplitAcrossWorkers` показывает лучший результат, потому что сжатие является CPU-bound задачей и четыре части файла обрабатываются параллельно. Накладные расходы TCP заметны, но меньше выигрыша от параллельного сжатия.

Режим `SingleWorker` нужен как распределенный аналог последовательной обработки. Он демонстрирует стоимость сетевого протокола и сериализации заголовков.

Режим `FullCopyToAllWorkers` не предназначен для ускорения одного файла. Он отправляет одинаковые данные всем воркерам и показывает разброс времени между узлами. Это удобно для сравнения производительности воркеров и проверки балансировки.

## Отказоустойчивость

При недоступности воркера heartbeat переводит его в состояние `нет heartbeat`. Если ошибка происходит во время обработки части, мастер повторно назначает часть другому здоровому узлу.

Ожидаемый результат теста отказа:

| Сценарий | Поведение системы |
| --- | --- |
| Воркер не отвечает на heartbeat | Узел исключается из выбора новых задач |
| Воркер падает во время сжатия | Часть переназначается другому воркеру |
| Все воркеры недоступны | Задача завершается ошибкой с сообщением |
| Пользователь нажимает отмену | Pipeline получает `CancellationToken`, задача получает статус `Отменена` |

## Вывод

Система масштабируется на CPU-bound задачах, когда файл можно разделить на независимые части. На локальном кластере из 4 воркеров режим разделения дает примерно 2.5-3x ускорение относительно последовательного сжатия, при этом сохраняет мониторинг, отмену и обработку отказов.

Фактические значения зависят от типа файла. Уже сжатые файлы, например `.zip`, `.jpg` или `.mp4`, могут дать меньший коэффициент сжатия и меньший выигрыш по времени.
