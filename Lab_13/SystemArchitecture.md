# Архитектура распределенной системы сжатия данных

## Назначение

Проект реализует распределенную систему сжатия файлов с веб-интерфейсом на Blazor. Пользователь загружает файл, выбирает алгоритм и режим распределения, после чего мастер-узел отправляет работу четырем локальным TCP-воркерам. Результат сохраняется в папку `CompressedFiles` в корне проекта.

## Основные компоненты

| Компонент | Файл | Ответственность |
| --- | --- | --- |
| Blazor UI | `MainPage.razor` | Загрузка файла, выбор режима, отображение прогресса и статистики |
| Точка входа | `Program.cs` | Настройка DI, запуск Blazor, регистрация endpoint для скачивания результата |
| Мастер-узел | `MasterCompressionService.cs` | Чтение файла, разбиение на задания, запуск пайплайна, агрегация результата |
| Воркеры | `WorkerNodeServer.cs` | TCP-серверы, принимающие команды `compress` и `heartbeat` |
| TCP-клиент | `TcpCompressionClient.cs` | Отправка заданий и heartbeat от мастера к воркерам |
| Протокол | `TcpProtocol.cs` | JSON-заголовок + бинарная полезная нагрузка |
| Реестр воркеров | `WorkerRegistry.cs` | Потокобезопасное состояние узлов, счетчики нагрузки и heartbeat |
| Пайплайн | `CompressionPipeline.cs` | TPL Dataflow-стадии подготовки, обработки и сбора результатов |
| Алгоритмы | `CompressionAlgorithms.cs` | GZip на `GZipStream` и ZIP на `ZipArchive` |
| Модели | `CompressionModels.cs` | DTO, статус задачи, прогресс воркеров, журнал операций |

## Мастер-рабочая архитектура

Мастер получает файл от Blazor UI и не сжимает его напрямую как единственный исполнитель. Он выбирает воркеров через стратегию `LeastLoadedWorkerSelectionStrategy`, формирует `CompressionWorkItem` и отправляет их по TCP. Воркеры работают независимо и возвращают сжатые байты, время CPU и статус выполнения.

В проекте запускается ровно 4 воркера:

- `worker-1`: `127.0.0.1:5101`
- `worker-2`: `127.0.0.1:5102`
- `worker-3`: `127.0.0.1:5103`
- `worker-4`: `127.0.0.1:5104`

Такой размер кластера выбран под 4-ядерный процессор пользователя.

## Режимы распределения

1. `SingleWorker`: весь файл отправляется одному наименее загруженному воркеру.
2. `SplitAcrossWorkers`: файл делится на части, каждая часть отправляется отдельному воркеру. Режим доступен только для GZip, результат сохраняется как WinRAR-совместимый multi-member `.gz`.
3. `FullCopyToAllWorkers`: весь файл отправляется всем четырем воркерам для сравнения параллельной обработки. В итоговый файл записывается самый быстрый полный результат, статистика сохраняется по каждому воркеру.

## Параллелизм и асинхронность

- TPL используется через `Task.Run`, `Task.WhenAll` и фоновые hosted-сервисы.
- `async/await` используется для чтения файла, TCP-обмена, записи результата и heartbeat.
- TPL Dataflow используется в `CompressionPipeline.cs`: `TransformBlock` готовит задания, второй `TransformBlock` выполняет сетевую отправку, `ActionBlock` собирает результаты.
- PLINQ используется в стратегии балансировки и при агрегации сетевого времени.
- `ConcurrentDictionary` хранит воркеров, задачи и прогресс.
- `BlockingCollection` используется для потокобезопасного журнала операций.
- `Interlocked` используется для счетчиков активных задач и завершенных частей.

## Мониторинг и отказоустойчивость

`WorkerMonitorHostedService` каждые 2 секунды отправляет heartbeat всем воркерам. Если воркер не отвечает, он помечается как недоступный и исключается из балансировки.

Если воркер падает или возвращает ошибку во время сжатия, мастер:

1. фиксирует ошибку в журнале задачи;
2. помечает воркер как проблемный;
3. выбирает нового здорового воркера;
4. повторно отправляет ту же часть файла.

Если здоровых воркеров больше нет, задача завершается ошибкой.

## Формат сетевого обмена

Каждое TCP-сообщение состоит из:

1. JSON-заголовка `TcpRequestEnvelope` или `TcpResponseEnvelope`, завершенного `\n`;
2. бинарного тела фиксированной длины `PayloadLength`.

Такой формат проще отлаживать, чем полностью бинарный протокол, но он не требует внешних библиотек и корректно передает произвольные файлы.

## Формат результата

- Для `SingleWorker` и `FullCopyToAllWorkers` сохраняется обычный `.gz` или `.zip`.
- Для `SplitAcrossWorkers` сохраняется multi-member `.gz`: каждый воркер возвращает независимый GZip-поток, а мастер склеивает потоки подряд в порядке частей. Первый GZip member содержит оригинальное имя файла в поле `FNAME`.

## Почему ZIP не делится на части

ZIP поддерживается WinRAR и стандартной библиотекой .NET, но один файл внутри ZIP должен быть представлен единым ZIP-entry. Независимые сжатые части, полученные от разных воркеров, нельзя просто склеить в один корректный ZIP-entry без пересжатия. Поэтому при выборе ZIP веб-интерфейс скрывает режим `Разделить файл на части`.

## Границы решения

Проект моделирует распределенный кластер локально через TCP-сокеты на `127.0.0.1`. Чтобы вынести воркеры на разные машины, достаточно запускать `WorkerNodeServer` на удаленных хостах и регистрировать их адреса в `WorkerRegistry`.
