# Руководство по реализации

## Выбор технологий

Проект написан на .NET 9 и Blazor Server/Razor Components. Такой выбор позволяет сделать веб-интерфейс без JavaScript-фреймворков, npm, Bootstrap или других внешних frontend-зависимостей. Вся страница находится в `MainPage.razor`, стили встроены в `App.razor`.

Для сжатия используются стандартные классы .NET:

- `GZipStream`
- `ZipArchive`

Для конкурентной обработки применяются:

- TPL и `Task` для фоновых операций;
- `async/await` для файлового и сетевого ввода-вывода;
- TPL Dataflow для асинхронного пайплайна;
- PLINQ для параллельных запросов к снимкам данных;
- `ConcurrentDictionary`, `ConcurrentQueue`, `BlockingCollection`;
- `Interlocked` для атомарных счетчиков;
- `CancellationToken` для отмены.

`System.Threading.Tasks.Dataflow` подключен как официальный пакет Microsoft, потому что API TPL Dataflow распространяется отдельно от базового shared runtime. Сторонние библиотеки в проекте не используются.

## Структура файлов

Все исходные файлы находятся в корне проекта:

- `Program.cs` — точка входа и настройка приложения.
- `App.razor`, `Routes.razor`, `MainPage.razor`, `_Imports.razor` — Blazor UI.
- `CompressionModels.cs` — DTO, статусы, задания и потокобезопасный журнал.
- `CompressionAlgorithms.cs` — реализации алгоритмов сжатия.
- `WorkerRegistry.cs` — реестр воркеров и стратегия балансировки.
- `TcpProtocol.cs` — протокол обмена.
- `TcpCompressionClient.cs` — клиент мастера для общения с воркерами.
- `WorkerNodeServer.cs` — TCP-сервер воркера.
- `WorkerHostedServices.cs` — запуск кластера и heartbeat-мониторинг.
- `CompressionPipeline.cs` — TPL Dataflow-пайплайн.
- `MasterCompressionService.cs` — основной сценарий сжатия.

## Запуск

1. Открыть терминал в корне проекта.
2. Выполнить восстановление зависимостей:

```
dotnet restore
```

3. Запустить приложение:

```
dotnet run
```

4. Открыть адрес, который выведет ASP.NET Core `http://localhost:5000`

## Алгоритм выполнения задачи

1. Пользователь выбирает файл, алгоритм и режим распределения.
2. Blazor читает файл через `OpenReadStream` и передает поток в `MasterCompressionService`.
3. Мастер копирует файл в память асинхронно и создает `CompressionJob`.
4. Мастер выбирает воркеров и готовит задания для выбранного режима.
5. В зависимости от режима мастер создает одно или несколько заданий.
6. `CompressionPipeline` отправляет задания в Dataflow-блоки.
7. `TcpCompressionClient` передает часть файла воркеру по TCP.
8. `WorkerNodeServer` сжимает данные и возвращает результат.
9. Мастер обновляет прогресс, агрегирует байты и сохраняет файл в `CompressedFiles`.

## Обработка ошибок

В проекте обрабатываются следующие случаи:

- пустой файл;
- отсутствие здоровых воркеров;
- ошибка TCP-соединения;
- ошибка сжатия на воркере;
- отмена пользователем;
- некорректный размер TCP payload;
- попытка скачать несуществующий результат.

При отказе воркера мастер не завершает задачу сразу. Он помечает узел как недоступный, выбирает замену и повторно отправляет часть файла. Если заменить воркер невозможно, задача завершается ошибкой.

## Масштабирование

Количество воркеров сейчас зафиксировано как 4 в `WorkerClusterHostedService`, потому что по заданию система должна работать под 4-ядерный процессор. Архитектурно реестр поддерживает добавление новых воркеров через `AddOrUpdate`, а стратегия балансировки уже работает по снимку всех здоровых узлов.

Чтобы изменить размер кластера, нужно поменять верхнюю границу цикла в `WorkerClusterHostedService.StartAsync`.

## Почему используется multi-member GZip

При режиме `SplitAcrossWorkers` каждая часть файла сжимается независимо. Для `GZip` стандарт разрешает multi-member формат: несколько GZip-потоков можно записать подряд, и совместимый архиватор распакует их как один файл. Поэтому разделенный `GZip` сохраняется как обычный `.gz`, а первый GZip member получает оригинальное имя файла в поле `FNAME`.

## Почему ZIP не поддерживает разделение

ZIP поддерживается WinRAR и есть в стандартной библиотеке .NET через `ZipArchive`. В проекте он используется для режимов `Один воркер` и `Отправить целиком всем воркерам`.

Для режима `Разделить файл на части` ZIP не подходит: чтобы внутри ZIP был один исходный файл, его содержимое должно быть одним корректным ZIP-entry. Такой entry использует единый поток сжатых данных, а не набор независимо сжатых частей от разных воркеров. Поэтому при выборе ZIP в интерфейсе пункт `Разделить файл на части` скрывается.

## Проверка корректности

Для ручной проверки:

1. Запустите приложение.
2. Загрузите файл.
3. Проверьте режим `Один воркер`.
4. Проверьте режим `Разделить файл на части`.
5. Проверьте режим `Отправить целиком всем воркерам`.
6. Убедитесь, что на странице появились прогресс-бары и статистика по каждому воркеру.
7. Убедитесь, что файл создан в `CompressedFiles`.
8. Нажмите отмену во время обработки большого файла и проверьте статус `Отменена`.

## Отказоустойчивость

Для демонстрации отказа можно остановить воркера. Heartbeat-монитор пометит узел как недоступный, а мастер попытается переназначить часть на другой воркер.

Invoke-RestMethod -Method Post http://localhost:5000/debug/workers/worker-1/stop
Invoke-RestMethod -Method Post http://localhost:5000/debug/workers/worker-1/start
